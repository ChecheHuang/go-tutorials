# 第四十二課：分散式一致性（Distributed Consistency）

> **一句話總結**：分散式系統不追求「絕對一致」，而是在一致性、可用性、效能之間做出正確的取捨。

## CAP 定理實戰

```
       C（一致性）
      ╱          ╲
     ╱    你只能    ╲
    ╱    選兩個      ╲
   ╱                  ╲
  A（可用性）─────── P（分區容錯）
```

| 系統 | 選擇 | 原因 |
|------|------|------|
| 搶票系統 | **AP** | 寧可庫存暫時不準（C），也不能讓系統掛掉（A） |
| 銀行轉帳 | **CP** | 寧可暫時不能用（A），也不能帳對不上（C） |
| DNS | **AP** | 寧可拿到舊紀錄（C），也不能解析失敗（A） |

## Inventory Token 流程

```
100 萬人同時請求
        ↓
  Token Bucket（只有 5000 個 token）
        ↓
  ┌─ 拿到 token（5000 人）→ 進入選座位頁面 → 鎖座位 → 付款
  └─ 沒拿到（99.5 萬人）→ 直接回傳「售罄」（不打 Redis）
```

**為什麼不直接讓 100 萬人選座位？**
- 100 萬次 Redis `SET seat:xxx NX` = Redis 被打爆
- 先發 5000 token = 只有 5000 次 Redis 操作

## 超賣問題

```go
// ❌ 不安全（race condition）
if stock > 0 {
    stock--  // 多個 goroutine 同時通過 >0 檢查
}

// ✅ 安全（原子操作，等同 Redis DECRBY）
if atomic.AddInt32(&stock, -1) >= 0 {
    // 成功扣庫存
} else {
    atomic.AddInt32(&stock, 1) // 回滾
}
```

## 2PC vs Saga

```
2PC（兩階段提交）：
  協調者 → 準備? → 庫存服務: OK ←
         → 準備? → 支付服務: OK ←
         → 提交!                     ← 全部一起提交
  問題：任何一方掛了，全部卡住

Saga（補償交易）：
  扣庫存 ✅ → 建訂單 ✅ → 支付 ✅ → 出票 ❌
                                      ↓
  回補庫存 ← 取消訂單 ← 退款 ← 補償開始
  結果：不會卡住，但需要寫補償邏輯
```

## 最終一致性 — 事件對帳

```
每 30 秒執行一次對帳：

  order-001: inventory ✅  payment ✅  order ✅  → 一致 ✅
  order-002: inventory ✅  payment ✅  order ❌  → 不一致！重建訂單
  order-003: inventory ✅  payment ❌  order ❌  → 不一致！退回庫存
```
